# Sklears v0.2.0 Release Notes

**Release Date**: July 14, 2026

## Overview

Sklears 0.2.0 rebuilds the project's GPU foundation on real `oxicuda` kernels â€” `sklears_core::gpu` now wraps genuine CUDA driver/BLAS/solver calls instead of placeholder scaffolding. Nine downstream crates (`sklears-clustering`, `sklears-cross-decomposition`, `sklears-decomposition`, `sklears-discriminant-analysis`, `sklears-linear`, `sklears-manifold`, `sklears-neighbors`, `sklears-neural`, `sklears-svm`) consume that foundation for on-device GEMM, solves, and elementwise kernels behind opt-in `gpu`/`gpu_support` features, with transparent CPU/Rayon fallback and honest `None` detection when no CUDA driver is present. This release also fixes several previously silent correctness bugs â€” including a hardcoded fake SVD and an input-ignoring t-SNE optimizer (see Notable Fixes below) â€” and re-enables `sklears-core` modules that had been fully excluded from the build. Across the workspace, 12,721/12,721 tests pass (100%, 161 skipped) across 36 crates; the default build remains pure Rust with zero C/Fortran dependencies, and WebGPU support is not yet implemented.

## Highlights

### GPU Foundation Rebuild (the headline change)

- `sklears_core::gpu` rebuilt on real `oxicuda` kernels: `GpuBackend` (pairs an `oxicuda_driver::Context` with an `oxicuda_blas::BlasHandle`), `GpuArray<T: Copy>` (backed by a real `oxicuda_memory::DeviceBuffer<T>`), and `GpuMatrixOps` (`matmul`/`add`/`mul`/`scale`/`transpose`) dispatching genuine on-device kernels.
- Detection is now honest: `GpuBackend::detect()`/`with_device_id()` return `Result<Option<Self>>` â€” `Ok(None)` when no GPU/CUDA driver is present, rather than silently substituting a CPU stub.
- This foundation is now the shared GPU layer for 9 downstream crates: `sklears-clustering`, `sklears-cross-decomposition`, `sklears-decomposition`, `sklears-discriminant-analysis`, `sklears-linear`, `sklears-manifold`, `sklears-neighbors`, `sklears-neural`, `sklears-svm` â€” each gained real on-device solves (an LDA generalized eigensolver, SVD/eigendecomposition, LU-solve/QR, a t-SNE optimizer, HNSW-backed approximate k-NN, FP16 tensor-core GEMM/conv2d, RBF/Sigmoid kernel transforms, and more), all behind opt-in `gpu`/`gpu_support` features with transparent CPU/Rayon fallback.
- GPU-context fields across those crates changed from eager/infallible to `Option`-wrapped, so construction now succeeds transparently on GPU-less machines instead of erroring.
- Dead/never-working GPU stub code removed: roughly 2,225 lines of always-failing stubs in `sklears-simd`, dead `Dummy*` GPU scaffolding in `sklears-cross-decomposition`, and a never-working `GpuKernel` trait in `sklears-ensemble`/`sklears-discriminant-analysis`.

### Re-enabled Modules

- `sklears-core::trait_explorer::graph_visualization` â€” re-enabled after being fully excluded from the build; 5 new `GraphAnalyzer` algorithms plus a new `api_reference_generator` module.
- `sklears-core::trait_explorer::security_analysis` â€” re-enabled with a real compliance/security-metrics pipeline (GDPR, HIPAA, CCPA, SOX, FERPA, ISO 27001, NIST CSF, COBIT, ITIL, CIS Controls).

- `sklears` facade crate â€” the `preprocessing` feature and `sklears-preprocessing` optional dependency restored, re-enabling 8 examples and 2 benchmark targets.
- `sklears-python` â€” `preprocessing` bindings (`StandardScaler`/`MinMaxScaler`/`LabelEncoder`) fully restored after being entirely disabled.

### New Capabilities

- `sklears-gaussian-process`: four previously stub modules fully implemented â€” `ConvolutionProcess`, `FitcGaussianProcessRegressor`, `KernelSelector`/`select_best_kernel`, `VariationalGaussianProcessClassifier`.
- `sklears-datasets`: 23 new public dataset generators across 6 new modules (`adversarial`, `causal`, `domain_specific`, `manifold`, `statistical`, `time_series`), with 100 new tests.

## Breaking Changes / Upgrade Guidance

The following changes require call-site updates when upgrading from 0.1.x. Each entry lists what changed and what to do about it.

### `sklears-core::gpu` (`gpu_support` feature)
**Changed**: `GpuContext::new()` and the old infallible `with_device_id() -> Result<Self>` are gone. `GpuContext` is now a type alias for `GpuBackend`, constructible only via fallible `detect()` / `with_device_id() -> Result<Option<Self>>`. `GpuArray<T>`'s bound loosened from `T: bytemuck::Pod` to `T: Copy` (`to_cpu`/`to_array2` now additionally require `T: Default`).
**Do this**: replace `GpuContext::new()` calls with `GpuBackend::detect()?` (or `with_device_id(id)?`) and handle the `Option` â€” `None` means no CUDA driver was found, so add a CPU fallback path instead of assuming a context always exists.

### `sklears-core::trait_explorer::graph_visualization`
**Changed**: `Graph3DGenerator::new` now takes `(ThreeDConfig, VisualizationTheme)`; `GraphExporter::available_formats()` renamed to `get_supported_formats()`; `Graph3DGenerator::generate_3d_scene()` renamed to `generate_3d_html()`; `LayoutAlgorithm` no longer derives `Copy, Eq, Hash` (its new `Custom` variant holds a `HashMap`).
**Do this**: update call sites to the new constructor signature and renamed methods; anywhere `LayoutAlgorithm` was copied or used as a `HashMap`/`HashSet` key, switch to `.clone()` or restructure around the dropped `Eq`/`Hash`.

### `sklears-cross-decomposition` (`gpu` feature)
**Changed**: the crate-local `GpuBackend` enum (which shadowed `sklears_core::gpu::GpuBackend`) is renamed to `GpuBackendKind`; `GpuAcceleratedContext::gpu_context() -> &dyn GpuContext` is replaced by `gpu_backend() -> Option<&GpuBackend>`.
**Do this**: rename local `GpuBackend` references to `GpuBackendKind`, and switch `gpu_context()` callers to `gpu_backend()`, adding `Option` handling.

### `sklears-discriminant-analysis` (`gpu` feature)
**Changed**: `GpuDiscriminantAnalysis::current_backend() -> Option<GpuBackend>` renamed to `backend() -> Option<&GpuBackend>`; `GpuAccelerationConfig` drops `preferred_backend` (the old `Cuda â†’ Metal â†’ Wgpu â†’ OpenCL â†’ Cpu` try-list is now a single CUDA-only `GpuBackend::detect()`) and `custom_kernels`.
**Do this**: rename `current_backend()` call sites to `backend()` and adjust for the borrowed return type; remove any code that set `preferred_backend`/`custom_kernels` on `GpuAccelerationConfig`.

### `sklears-compose`
**Changed**: `ModelFusion`'s and `HierarchicalComposition`'s `Trained` state each changed from a bare type alias (e.g. `type ModelFusionTrained = ModelFusion<Trained>`) to a dedicated struct holding the genuinely-fitted base models.
**Do this**: code that relied on the old type-alias identity (e.g. constructing a `Trained` value directly rather than via `.fit()`) needs to go through the new struct's constructor/`fit()` path instead.

### `sklears-python`
**Changed**: `get_hardware_info()`'s return type changed from `HashMap<String, bool>` to a heterogeneous Python `dict` (`PyResult<Py<PyDict>>`), since the CPU core count needed to carry a real integer rather than being coerced into a boolean; `show_versions()` now returns `PyResult<String>`.
**Do this**: Python callers reading values out of `get_hardware_info()` must expect mixed value types (ints/strings/bools) in the returned dict rather than assuming every value is a bool; callers of `show_versions()` can now capture its returned version string.

## Notable Fixes

A representative sample of the most significant correctness fixes in this release â€” cases where code previously produced silently wrong or fabricated results. See `CHANGELOG.md` for the complete list.

- **`sklears-decomposition`**: `ParallelDecomposition::sequential_svd`/`block_parallel_svd` returned a **hardcoded `(eye(m), ones(min_dim), eye(n))`** for every input regardless of the actual data. Both now compute a real SVD, verified by new regression tests that reconstruct the input matrix from the decomposition.
- **`sklears-manifold`**: `GpuTSNE::fit_transform` ignored its input entirely and returned a random-normal embedding as though it had converged. It now runs the real `oxicuda_manifold::tsne_fit` optimizer (and, without the `gpu` feature, honestly returns `Err(MissingDependency)` instead of the same fake output).
- **`sklears-covariance`**: `GraphicalLasso`'s coordinate-descent solver had an exact even/odd `max_iter` parity bug â€” its update rule formed a period-2 cycle from a zero seed, so every off-diagonal entry landed back at exactly `0` at the end of any even `max_iter` (including the crate's own default of 100), silently degenerating every fit to the identity matrix regardless of `alpha` or the input data. Replaced with a real block coordinate-descent implementation (Friedman/Hastie/Tibshirani 2008).
- **`sklears-compose`**: `ModelFusion::fit()`'s "train all base models" step was a silent no-op â€” it looped over the configured base models and pushed each one back unfitted. It now genuinely fits every base model (only `FusionStrategy::NeuralNetwork` has a full training implementation so far; every other strategy now returns an honest `Err(NotImplemented)` instead of fabricating a "successfully trained" result).
- **`sklears-gaussian-process`**: `GaussianProcessRegressor::predict_with_std` computed the predictive-variance quadratic form as `v_i Â· v_i` instead of `k_star_i Â· v_i`, silently corrupting the predictive standard deviation/variance of the crate's core reference regressor.
